- 状态：已确认设计，待实施

- 目标形态：原生 HTML 多页面中英双语静态网站

## 1. 项目概述

本项目建设一个长期维护的个人专业网站，用于完整呈现徐建硕在财产保险领域的职业经历、专业能力、代表项目和工作思考。网站不是刻意的求职落地页，也不使用“正在求职”等直接话术；它首先是个人专业档案和工作知识沉淀空间，同时让招聘方、行业同仁和合作伙伴能够自然了解其经验与能力。

网站采用原生 HTML、CSS 和少量 JavaScript，实现内容丰富、加载快速、部署简单、可直接绑定个人域名的多页面静态站。中文是内容原稿，英文是对应的专业化改写版本。

## 2. 目标与成功标准

### 2.1 核心目标

1. 在首页快速建立清晰、可信的专业形象。
2. 系统呈现十年财产险从业经验及跨职能管理能力。
3. 以结构化项目总结体现分析方法、推动能力和业务成果，而不是简单重复简历。
4. 建立可长期更新的专业文章与项目档案。
5. 支持中英文访问、独立域名、搜索引擎收录和移动端阅读。

### 2.2 成功标准

- 访问者在首页首屏能识别姓名、专业领域、核心资质和经验年限。
- 访问者在两次点击内可到达任一职业经历、项目总结或专业文章。
- 中文页面均能切换到对应英文页面，英文页面也能返回对应中文页面。
- 手机、平板和桌面端均无需横向滚动，正文具备舒适的阅读宽度。
- 关闭 JavaScript 后，导航和主要正文仍可访问。
- 公开内容不包含完整手机号、出生年月、内部规则或未经确认的敏感经营数据。

## 3. 目标受众

### 3.1 主要受众

- 财产险行业管理者及专业人士
- 保险公司、保险科技平台及相关机构的招聘与业务负责人
- 精算、产品、核保、风控、渠道和保险经营领域的同行

### 3.2 访问者希望获得的信息

- 徐建硕是谁，主要从事什么工作
- 其能力覆盖哪些保险经营与专业领域
- 在复杂项目中承担什么职责、采用什么方法、产生什么成果
- 对行业问题和日常工作的思考是否系统、可信、可复用
- 如何建立联系

## 4. 定位与内容语气

### 4.1 推荐定位

中文：**财产保险经营、精算与风险管理实践者**

英文：**Property Insurance, Actuarial & Risk Management Practitioner**

该定位覆盖保险经营、精算定价、产品开发、核保风控、渠道平台和系统规划等经历，不把个人形象限制在单一岗位。

### 4.2 语气原则

- 专业但不堆砌术语。
- 用事实、方法和结果建立可信度，不使用空泛口号。
- 不刻意求职，不设置“Hire Me”等强求职按钮。
- 第一人称内容用于个人简介与文章；职业经历和资质信息保持客观简洁。
- 英文内容按英语专业表达习惯改写，不逐字翻译中文。

## 6. 页面设计

### 6.1 首页

首页承担专业定位和内容分发功能，建议依次包含：

1. **首屏**：姓名、专业定位、十年财产险经验简介、所在地，以及 FCAA、FRM、CPA、深圳市高层次人才等核心资质。
2. **专业能力全景**：保险经营、精算定价、产品开发、核保风控、渠道平台、系统规划六个能力模块。
3. **精选职业成果**：使用简短事实和已经确认的数字呈现代表性成果。
4. **精选项目**：展示三至五个项目卡片，进入项目详情页。
5. **最新专业文章**：展示最近更新的文章摘要。
6. **职业经历摘要**：以精简时间线连接到完整经历页。
7. **联系区**：公开邮箱、脱敏手机号和 GitHub 链接。

首页内容充实，但通过明确分区、留白和“查看详情”控制信息密度。

### 6.2 职业经历页

按时间倒序展示工作经历。每段经历由公司、部门、职位、时间、职责范围和代表成果组成。职责按业务主题分组，避免逐条照搬简历。

教育背景展示湖南大学金融学硕士、保险学本科及就读时间。资质区展示中国精算师（FCAA）、金融风险管理师（FRM）、注册会计师（CPA）和深圳市高层次人才。

### 6.3 项目列表页

项目卡片包含标题、类别、时间、摘要、个人角色和详情入口。可通过少量 JavaScript 按以下主题筛选：

- 车险经营与续保
- 精算定价与成本管理
- 产品开发
- 核保与风险管理
- 渠道及平台合作
- 系统规划与流程优化

筛选仅改善浏览体验；关闭 JavaScript 时仍显示全部项目。

### 6.4 项目详情页

每篇项目总结使用统一结构：

1. 项目背景
2. 核心问题
3. 个人职责
4. 分析框架与方法
5. 实施过程
6. 结果与影响
7. 复盘与可复用经验

首批内容计划包括：

- 意外险新规整改项目
- 个人非车险品质优化项目
- 车险续保经营与流程优化
- 第三方合作平台对接与运营
- 非车险成本管理体系建设

公开版本不披露内部阈值、具体核保规则、未公开合作细节或可识别的客户信息。

### 6.5 专业思考页

文章列表按保险经营、精算与数据、产品方法、管理复盘分类。文章详情页包含标题、发布日期、摘要、正文、相关主题和上一篇/下一篇导航。项目总结强调“做过什么”，专业文章强调“如何理解问题”。

### 6.6 关于页

使用较自然的第一人称介绍职业路径、工作方法和长期关注主题。联系信息只包含公开邮箱、脱敏手机号和 GitHub，不展示出生年月。

## 7. 初始内容映射

### 7.1 核心能力

- **保险经营**：经营分析、目标调度、业绩监控、预算与 KPI 管理。

## 8. 视觉系统

### 8.1 风格

采用“专业克制 + 编辑杂志感”的混合方向。整体稳健、清晰、有阅读品质，不使用传统在线简历模板式的大量进度条、技能百分比或夸张动效。

### 8.2 色彩

- 主色：深海军蓝，用于标题、导航和主要文字。
- 背景：暖白，降低长文阅读疲劳。
- 强调色：低饱和金色，仅用于资质、关键数字和交互状态。
- 辅助色：蓝灰，用于次级文字、边框和标签。

最终实现时以 CSS 自定义属性统一管理，并保证正文及交互元素满足 WCAG AA 对比度要求。

### 8.3 字体与排版

- 中文正文使用系统无衬线字体栈，确保加载速度和跨设备稳定性。
- 中文标题可使用系统宋体或衬线字体增强编辑感。
- 英文标题与正文分别使用对应 Serif 与 Sans 字体栈。
- 长文正文限制最大行宽，采用舒适行高；移动端字号不小于 16px。

### 8.4 关键界面元素

- 固定顶部导航与清晰的当前栏目状态。
- 中英文切换器跳转到对应页面，而非仅替换局部文字。
- 资质标签、能力模块、项目卡片、文章卡片和职业时间线使用统一视觉语言。
- 动效限定为淡入、轻微位移和悬停反馈，并尊重 `prefers-reduced-motion`。

## 9. 技术架构

### 9.1 技术选择

- 语义化 HTML5
- 原生 CSS
- 少量原生 JavaScript
- 无前端框架、无包管理器、无构建步骤

### 9.3 组件边界

- **页面框架**：页头、主导航、页脚、语言切换。
- **内容组件**：资质标签、能力卡片、项目卡片、文章卡片、时间线、数字事实块。
- **交互组件**：移动导航、项目筛选、轻量进入动画。
- **内容页面**：首页、经历、项目、文章、关于；共享视觉组件但保持独立 HTML。

组件使用一致的 HTML 类名和可访问性约定。任何交互组件都不得成为阅读内容的唯一入口。

### 9.4 双语路由

中文位于根路径，英文位于 `/en/`。每个页面设置：

- 对应语言的 `lang` 属性
- 指向自身的 canonical URL
- 中文、英文及默认语言的 `hreflang` 链接
- 指向对应语言页面的显式切换链接

首版公开内容必须同时具备中文和英文页面。新文章可以先在本地完成中文草稿，但只有对应英文版本完成后才加入公开列表和站点地图，从而保证语言切换始终到达同一内容的对应版本。

## 10. 数据流与更新方式

网站没有后台、数据库或远程内容接口。内容直接存在各 HTML 文件中：

1. 作者复制对应的项目或文章 HTML 模板。
2. 填写正文、摘要、日期、分类和页面元信息。
3. 在列表页增加对应卡片。
4. 同步新增或更新英文页面。
5. 更新 `sitemap.xml`。
6. 本地检查后提交到 GitHub，由静态托管平台发布。

这一方式符合原生 HTML 偏好，也避免首版引入内容管理系统。若文章量明显增加，可在后续版本评估静态站点生成器，但不属于当前范围。

## 11. SEO 与分享

- 每页包含独立标题、描述、canonical 和 Open Graph 元信息。
- 使用语义化标题层级、正文结构和内部链接。
- 首页提供 `Person` 结构化数据，文章页提供 `Article` 结构化数据。
- 提供 `sitemap.xml` 与 `robots.txt`。
- URL 使用简短、稳定的英文 slug。
- 绑定域名后，所有绝对地址统一更新为正式 HTTPS 域名。

## 13. 降级与错误处理

- 提供自定义 `404.html`，包含返回首页和主要栏目入口。
- JavaScript 加载失败时，正文、普通导航和全部项目仍可访问。
- 图片加载失败时保留替代文本和稳定布局，不影响正文阅读。
- 项目筛选无结果时显示明确提示和“查看全部”操作。
- 外部链接使用清晰标签；新窗口链接采用安全的 `rel` 属性。
- 域名未配置或 HTTPS 证书尚未生效时，不提前发布正式域名链接。

## 14. 测试与验收

### 14.1 功能检查

- 所有导航、项目、文章和中英文对应链接有效。
- 项目筛选、移动导航和语言切换可用。
- 404 页面能引导用户返回有效页面。
- 关闭 JavaScript 后主要内容可读。

### 14.2 视觉与响应式检查

- 在常见桌面、平板和手机宽度下检查布局。
- 页面无横向滚动、文字遮挡或点击区域重叠。
- 长文章阅读宽度、字号和行高合适。
- 动效关闭设置得到尊重。

### 14.3 质量检查

- 验证 HTML 结构和标题层级。
- 检查键盘导航、焦点状态、颜色对比和图片替代文本。
- 检查页面标题、描述、canonical、`hreflang`、结构化数据和站点地图。
- 检查图片尺寸、延迟加载和首屏资源。
- 搜索敏感信息并人工复核项目数据口径。

## 15. 部署与域名

网站适配 GitHub Pages 或其他支持 Git 仓库的静态托管平台。实施阶段根据域名服务商和托管偏好确定具体平台。部署要求如下：

- 从 GitHub 默认发布分支持续部署。
- 绑定本人申请的自定义域名。
- 强制 HTTPS。
- 配置根域名与 `www` 的统一跳转策略。
- 上线前更新 canonical、Open Graph、结构化数据和站点地图中的正式域名。
